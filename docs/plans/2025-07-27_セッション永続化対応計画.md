# セッション永続化対応計画

**作成日**: 2025年7月27日  
**最終更新**: 2025年7月27日  
**対象**: 農業AIエージェントシステム  
**目的**: Redis導入による会話履歴の永続化

## 概要

エージェント文脈保持システムの実装完了に伴い、現在はインメモリ履歴を使用している。本番環境でのユーザー体験向上のため、Redis導入による永続的な会話履歴管理を計画する。

## 現在の状況

### 実装完了事項（2025-07-27）
✅ **エージェント文脈保持システム完成**  
✅ **SessionManager**: Redis接続失敗時のインメモリフォールバック  
✅ **ConfirmationMiddleware**: 確認フロー状態管理  
✅ **MasterAgent**: LangChain Memory統合（フォールバック付き）  
✅ **WorkLogRegistrationAgentTool**: 確認フロー対応  

### 現在の動作状況
- **会話履歴**: インメモリ（ChatMessageHistory）
- **セッション管理**: SessionManager（インメモリフォールバック）
- **全機能**: 正常動作中
- **制限事項**: サーバー再起動で会話履歴リセット

## 背景・課題

### 開発環境での制限
- プロセス終了やサーバー再起動でセッションが消失
- マルチワーカー構成ではセッションが各プロセスに分散

### 本番環境での要求
1. エージェントの確認フローや会話履歴を永続化し、途中でサーバーが落ちても復旧できるようにする
2. 本番環境・複数ワーカー環境でも安定してセッション共有を行う

## Redis導入の必要性・メリット

### 1. 24時間持続する会話記憶
```
農家: 「昨日話したトマトハウスの件」
↓ （翌日でも文脈理解）
Bot: 「昨日のトマトハウスの薬剤散布の件ですね」
```

### 2. サーバー再起動耐性
- システムメンテナンス時も会話履歴保持
- ユーザー体験の継続性確保
- 運用安定性の向上

### 3. スケーラビリティ対応
- 複数サーバーインスタンス間での会話履歴共有
- 将来的な負荷分散対応

## 技術方針

| 優先度 | 選択肢 | 概要 |
|--------|--------|------|
| ★★★   | Redis  | 低レイテンシ、シンプルなキー値ストア。LangChain 公式実装あり。 |
| ★★    | MongoDB | 既存 DB を流用。複雑だが運用コスト低。 |
| ★     | DynamoDB / Cloud KV | 将来のクラウド移行を想定。|

まずは **Redis** を優先採用し、フォールバックとしてインメモリを残す。

## Redis環境構築

### 開発環境
```bash

brew install redis
brew services start redis

# Docker
docker run -d -p 6379:6379 redis:alpine
```

### 本番環境オプション
1. **自前サーバー**: Redis単体サーバー
2. **クラウドサービス**: 
   - AWS ElastiCache
   - Google Cloud Memorystore
   - Azure Cache for Redis

## 技術仕様

### 現在の実装アーキテクチャ

```python
# Redis接続成功時
message_history = RedisChatMessageHistory(
    session_id=f"{user_id}:{thread_id}",
    url="redis://localhost:6379",
    ttl=86400  # 24時間
)

# Redis接続失敗時（フォールバック）
message_history = ChatMessageHistory()  # インメモリ
```

### メモリ管理

| 機能 | Redis接続時 | フォールバック時 |
|------|-------------|------------------|
| 会話履歴 | 24時間永続 | セッション内のみ |
| 確認フロー状態 | 30分永続 | インメモリ管理 |
| エージェント機能 | 全機能利用可能 | 全機能利用可能 |
